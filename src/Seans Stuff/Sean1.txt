Beginner guide for teammate: ROS 2 web dashboard and autonomy foundation
What this project is

We are building the foundation for a Lunabot control and monitoring system.

The goal is to let a normal web browser talk to the robot’s ROS 2 system in real time.

That means:

the browser can read robot data
the browser can send robot commands
the browser can call ROS services
later, the browser can become our autonomy dashboard

This is useful because it gives us one place to monitor and control the robot during development and testing.

The big picture

Think of the system like this:

Browser dashboard
      ↓
JavaScript (roslibjs)
      ↓
WebSocket connection
      ↓
rosbridge_server
      ↓
ROS 2 graph
      ↓
Robot topics, services, and nodes
What each part means

Browser dashboard
This is the webpage we open in Chrome or Edge. It is our user interface.

JavaScript / roslibjs
This is the code in the webpage that knows how to talk to ROS.

WebSocket connection
This is the live communication line between the browser and ROS.

rosbridge_server
This is the translator. Browsers do not understand native ROS communication, so rosbridge_server converts ROS messages into a format the browser can use.

ROS 2 graph
This is the network of ROS topics, services, and nodes running on the robot or laptop.

What we have already built

So far, the system already does these things:

1. Browser connects to ROS 2

We used rosbridge_server and roslibjs to make a webpage connect to ROS through port 9090.

2. Dashboard runs locally

We serve the dashboard using a simple Python web server on port 8080.

That means:

webpage is at http://localhost:8080
ROS bridge is at ws://localhost:9090
3. Subscribe to topics

We already proved the browser can listen to ROS topics.

Examples:

/rosout
/odom
4. Publish to topics

We already proved the browser can publish to:

/cmd_vel

using buttons like Forward, Back, Left, Right, Stop.

5. Call services

We already proved the browser can call a ROS service:

/rosapi/topics

to ask ROS which topics are active.

6. Display odometry live

We built a fake odometry publisher and connected it to the dashboard, so the dashboard now shows live X and Y position.

Why this matters for autonomy

This work does not make the robot autonomous by itself.

What it does is create the interface layer for autonomy.

Autonomy needs these pieces:

sensor data
robot position
mapping
planning
control
state monitoring
operator supervision

The dashboard we built becomes the place where we can:

see what the robot thinks
see sensor data
send commands
start and stop autonomy
debug failures in real time

Without a dashboard, autonomy is mostly invisible.
With a dashboard, we can understand what the robot is doing.

Part 1: environment setup
Important concept: Windows vs Ubuntu

This project is being developed on a Windows laptop, but ROS 2 is running inside Ubuntu through WSL.

That means there are two environments:

Windows / PowerShell

Looks like this:

PS C:\Users\cards>

This is not where we run ROS commands.

Ubuntu / WSL

Looks like this:

sean@LAPTOP-xxxx:~$

This is where we run ROS commands.

If a command uses things like:

source
/mnt/c/...
ros2
python3

it should probably be run in Ubuntu, not PowerShell.

Opening Ubuntu
Click Start
Search for Ubuntu
Open it

You should see a prompt like:

sean@LAPTOP-xxxx:~$

This is a very important strategy:

Build the software pipeline first
Swap fake data for real data later

That is exactly what strong robotics teams do.

Part 6: what teammate should learn first

The teammate should get comfortable with four basic ideas.

1. Topics

Topics are streams of data.

Examples:

/odom
/cmd_vel
/scan

A topic is like a radio station:

one node publishes
another node listens
2. Services

Services are request/response.

Example:

ask ROS for all active topics

A service is like asking a question and waiting for one answer.

3. Nodes

Nodes are programs inside ROS.

Example:

fake odom publisher
rosbridge server
future encoder node
future LiDAR node
4. Messages

Messages are the data types passed around in topics and services.

Examples:

nav_msgs/msg/Odometry
geometry_msgs/msg/Twist
sensor_msgs/msg/LaserScan

Part 8: what partner should understand about ports

This is very important.

Port 8080

This is the webpage server.

You type this into the browser:

http://localhost:8080
Port 9090

This is rosbridge.

The browser does not open this directly.
Instead, the JavaScript inside the webpage connects to it using:

ws://localhost:9090

So:

browser loads page from 8080
webpage JS talks to ROS through 9090

Part 12: common mistakes
Mistake 1: using PowerShell instead of Ubuntu

Wrong prompt:

PS C:\Users\cards>

Use Ubuntu instead.

Mistake 2: typing a URL into the terminal

This is wrong:

http://localhost:8080

That goes in the browser, not the terminal.

Mistake 3: opening multiple rosbridge servers

If port 9090 is already in use, another rosbridge is probably still running.

Mistake 4: using Ctrl+Z

This suspends a process instead of stopping it cleanly.

Prefer:

Ctrl + C
Mistake 5: forgetting to hard refresh

If app.js changes, use:

Ctrl + Shift + R
Part 13: what he should say if asked what this system does

A good plain-English explanation is:

We built a live web dashboard that can talk to ROS 2 in real time. It can read robot data, send commands like velocity messages, and call ROS services. Right now it uses fake odometry because the real robot is not finished, but the idea is that later we replace the fake data with real sensors and keep the rest of the software pipeline the same.

That’s the core idea.
